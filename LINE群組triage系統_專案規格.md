# LINE 工作群組健康度 Triage 系統 — 專案規格

> 本文件供 AI agent 作為實作依據。內容已收斂前期設計討論的關鍵決策，請勿在實作中推翻這些決策（若要調整需先回報）。標示「待定」的項目需向專案負責人確認後再進行。

---

## 1. 專案概述

**目標**：建立一套量化指標系統，讓主管/老闆在同時管理「上百個」LINE 工作群組時，**不必逐一閱讀**就能知道哪些群組需要優先介入處理。

**情境**：
- 群組為公司內部工作群，**客戶在群組內**，由公司員工在群內處理客戶問題。
- 資料為繁體中文（台灣語境），夾雜英文、台語、emoji、貼圖、碎片化短訊。

**使用者**：主管/老闆。輸出為一份**依關注優先度排序**的群組清單 + 可下鑽的佐證。

**核心定位（重要）**：這是一個 **triage / ranking 問題**，不是逐群正常與否的分類問題。系統只要讓「最該被看的群」浮到最前面即可。因此：
- 主要優化目標為 **precision@K**（排在前 K 名的群是否真的值得介入），而非整體分類準確率。
- 輸出為連續分數 + 排序 + 紅旗，而非二元標籤。

---

## 2. 範圍與限制

**已確認的前提條件**：
- 客戶在群組內 → 員工 vs 客戶的角色可由**群成員 metadata** 辨識（員工為已知公司帳號），不需 NLP 推斷。
- **無法取得**「主管事後關注名單」這類回溯標籤 → 不走純監督式學權重起步，採用「營運門檻 + 紅旗 + 前瞻回饋迴路」策略（見 §6、§7）。

**Out of scope（初期）**：
- 完整的對話解纏（conversation disentanglement）。優先利用 LINE 的「回覆」功能取得 thread link；無 reply link 時用時間窗近似，不做重型解纏模型。
- 自動回覆客戶 / 自動處理。系統只做「偵測與排序」，不介入對話。

---

## 3. 資料擷取與前處理

### 3.1 資料來源（待定，需確認）
擷取路徑二選一，需向專案負責人確認可行性：
- (A) LINE Messaging API / 官方帳號 bot 加入群組接收訊息（即時，但 bot 需為群成員，且有平台限制）。
- (B) 群組對話匯出檔批次處理（離線，較易取得但非即時）。

建議先以 (B) 跑通離線 pipeline 與指標驗證，再評估 (A) 做即時化。

### 3.2 角色辨識
- 維護「員工帳號清單」→ 訊息發話者比對成員身分，標記 `role ∈ {staff, customer}`。
- 此為多數延遲指標的前提，務必先做且要準。

### 3.3 訊息正規化
- **碎片合併**：同一發話者在短時間（如 < 60 秒，可調）內連續送出的多則短訊，合併為一則邏輯訊息。
- **繁中處理**：斷詞 / POS / NER 使用 **CKIP Transformers（中研院）**，對台灣語境與繁體較貼。
- **emoji / 貼圖**：保留並映射為語意/情緒訊號（如 👍 可作為確認/閉合訊號），不可直接丟棄。
- **thread link**：若訊息使用 LINE 回覆功能，記錄被回覆訊息 id，供問答配對使用。

### 3.4 訊息層級欄位（enrichment，逐則訊息產出）
| 欄位 | 說明 |
|---|---|
| `group_id`, `msg_id`, `sender_id`, `role`, `timestamp` | 基本欄位 |
| `reply_to_msg_id` | 若使用回覆功能 |
| `dialogue_act` | question / request / answer / ack / report / chit_chat（可先用規則 + LLM 輔助） |
| `sentiment` | 情緒分數（客戶端為重點） |
| `is_escalation_marker` | 是否含升級詞（見 §5） |
| `embedding` | 句向量（供語意指標用，多語 e5 / BGE-m3 或商用 embedding） |

## 4. 指標體系（Index System）

設計原則：指標間盡量正交、各自映射為 `[0,1]` 嚴重度（severity）後再合成。**兩種正規化方式分流**：

- **絕對營運門檻**（適用有天然門檻的指標）：讓系統能誠實輸出「今天無群需關注」或「今天多群在燒」，避免相對排名永遠製造一個「最差 5%」。
- **對該群 baseline 的 z-score**（適用無天然門檻的動態指標）。

所有與「議題年齡」相關的成分需加**時間衰減**，只反映「此刻仍卡著」的程度，已解決的不應持續掛紅。延遲計算需**業務時段感知**（service-hour aware），非工時不累計（服務時段為待定參數）。

### 4.1 絕對門檻類指標（優先實作，皇冠指標）

**I1 — 未回應提問存活時間 (Unanswered Question Age)** ★最高優先
- 對每則 `role=customer` 且 `dialogue_act∈{question,request}` 的訊息，尋找其後首則 staff 回應（優先用 reply link，否則同群任一 staff 訊息）。
- 若無回應 → `age = now - timestamp`（業務時段內計）。
- 指標值 = 該群所有開放提問中 severity 的最大值。
- severity 映射（門檻可調，預設）：`<30min→0`；`30min–6h` 漸增；`>6h→1`。

**I2 — 客戶回應延遲 P90 (Customer Response Latency)**
- 滾動窗內已回應提問的 `(首則 staff 回應時間 − 客戶提問時間)` 分布，取 P90。
- severity 由門檻映射（預設 P90 `>2h` 開始升、`>8h→1`，可調）。

**I3 — 未解決議題最老年齡 (Oldest Open Issue Age)**
- 來源為 §6 LLM 議題抽取中 `status=unresolved` 的議題。
- severity 由最老未解議題年齡映射，加時間衰減。

**I4 — 客戶負面情緒 (Customer Negative Sentiment)**
- 近窗 `role=customer` 訊息中負面比例 + 突增成分。
- severity 由負面比例門檻映射；可疊加對 baseline 的 spike 偵測。

### 4.2 Baseline 相對類指標（次要 / 輔助）

**I5 — 量能異常 (Volume Anomaly)**：訊息率對該群滾動 baseline 的 z-score（爆量常為救火）。

**I6 — 反常靜默 (Abnormal Silence)**：原本活躍的群忽然沉默且仍有開放議題 → 升 severity。

**I7 — 語意失序 (Semantic Disorder)** ☆最低優先、可選
- entropy / 相鄰訊息 coherence 對 baseline 的 z-score。
- 註：此為研究味訊號，可行動性低、特異性弱（群聊本就高 entropy），列為輔助 channel，**不可當主力**。先把 I1–I4 做穩再考慮。

---

## 5. Tripwire 紅旗層（非補償式覆蓋）

獨立於加權分數之外。任一條件觸發 → 該群**強制置頂**，不參與平均（避免「一個客戶氣到要退費」被其他正常指標稀釋）。門檻用業務常識直接設，不需標籤。

觸發條件（清單可調）：
- 升級 / 危機詞：`退費`、`投訴`、`換人`、`找你主管`、`找你老闆`、明示要離開/解約。
- 任一客戶提問未回應超過硬上限（如 8 業務小時）。
- 主管/老闆帳號被 `@tag` 或主動加入該群（同時作為 §7 的天然 weak label）。

---

## 6. LLM 議題抽取層

對話切段後，由 LLM 產出結構化議題清單，供 I3 與「可解釋佐證」使用。**成本控制：分層執行**——便宜統計指標（I1、I2、I4、I5）對所有群持續跑；LLM 抽取只**定期**跑、或只在便宜指標/紅旗觸發的群上跑。

輸出 schema（每議題一筆）：
```json
{
  "issue_id": "string",
  "group_id": "string",
  "raised_by": "customer|staff",
  "raised_at": "timestamp",
  "summary": "一句話描述問題",
  "type": "question|request|complaint|report",
  "status": "resolved|unresolved|unclear",
  "resolution_evidence": "閉合佐證訊息 id 或說明（道謝/確認/👍）；無則 null",
  "last_activity_at": "timestamp",
  "evidence_msg_ids": ["供下鑽用"]
}
```
進階（可選）：對「此議題解決了嗎」讓 LLM sample 多次估 semantic entropy，高不確定 → 標 `unclear` 拉出人工複核。

## 7. 合成分數與權重策略

**雙軌輸出**：
1. **連續健康度分數**：`composite = Σ wᵢ · severityᵢ`（I1–I7 正規化後加權和）。
2. **紅旗覆蓋**：任一 tripwire 觸發 → 強制最高優先層，連續分數僅用於同層內排序。

**權重來源（因無回溯標籤）**：
- 初期：**手設權重，但每個權重需講得出營運理由**（如 I1 權重最高）。錨在營運常識，非憑空調參。
- 過渡（可選）：若要資料驅動，可用 PCA/因子載荷或**熵權法**起步，但須註明其優化的是區分度而非「關注相關性」，僅為啟發式。

**前瞻回饋迴路（關鍵，使系統可自我升級）**：
- 儀表板在每日 top-K 名單上提供主管 `👍/👎`「是否值得看」，或兩兩比較「A 是否比 B 急」（對應 learning-to-rank）。
- 累積到一定量後，用 logistic / ranking 模型學權重取代手設值——學出的線性模型本身即「加權加總成單一 index」，但具校準、可驗證。

**Weak / proxy 標籤（供驗證與 bootstrap）**：主管被 tag/加入群、事後客訴/流失/退費（若 CRM 可 join）、議題是否閉合。雜訊大但可用於 sanity check 排序，或 Snorkel 式合成機率標籤。

---

## 8. 系統架構

```
擷取 → 訊息正規化/enrichment（角色、dialogue act、情緒、embedding）
   → [便宜層：持續] 統計指標 I1,I2,I4,I5,I6 + Tripwire
   → [貴層：觸發/定期] LLM 議題抽取 → I3,(I7)
   → 合成分數 + 紅旗覆蓋（加時間衰減）
   → 排序儀表板（top-K + 下鑽佐證 + 回饋 widget）
   → 回饋資料 → (累積後) 學權重，回灌
```

**儲存**：訊息層 enriched store + 群層指標時間序列（供 baseline 與趨勢）。
**處理模式**：先批次跑通，再評估即時化。

---

## 9. 評估方案（無標籤起步）

- **Face validity 複核**：人工檢視 top-K 與 bottom-K，確認排序合理。
- **Proxy-label 相關性**：檢查高分群是否較常對應下游升級/客訴/流失。
- **回饋累積後**：用 👍/👎 與兩兩比較計算 NDCG / precision@K，作為正式排序指標。

---

## 10. 隱私與合規

客戶在群內 → 處理客戶個資。需先釐清台灣**個資法（PDPA）**下的蒐集/處理目的、告知與同意（員工與客戶），勿等系統完成才處理。資料最小化、存取控管、保存期限需納入設計。

---

## 11. 分階段路線圖

**Phase 0 — 資料與前處理**
資料擷取打通、角色辨識、訊息正規化、enrichment 欄位產出。

**Phase 1 — v1（可上線、不需 LLM 核心）**
I1–I2、I4 + Tripwire + 手設加權合成 + 基本排序儀表板（含 👍/👎 回饋 widget）。目標：先讓皇冠指標 + 紅旗產生價值，並開始收回饋。

**Phase 2 — v2**
加入 LLM 議題抽取層（分層觸發）、I3 未解議題指標、下鑽佐證。

**Phase 3 — v3**
回饋累積 → learning-to-rank 取代手設權重；加入 weak-label 驗證；I7 語意指標作為輔助 channel 試驗。

---

## 12. 技術選型建議

| 項目 | 建議 |
|---|---|
| 繁中斷詞/NER/POS | CKIP Transformers（中研院） |
| 句向量 embedding | multilingual-e5 / BGE-m3 或商用 embedding |
| 情緒分析 | 中文情緒模型或 LLM 標註 |
| 議題抽取 | 具 structured output 能力的 LLM |
| 後端/排程 | 任意（批次先行） |
| 儀表板 | 排序清單 + 下鑽 + 回饋 widget |

---

## 13. 需向專案負責人確認的決策點

1. **資料擷取路徑**：LINE 官方帳號 bot 入群（即時）vs 對話匯出批次（離線）。
2. **服務時段定義**：業務時段感知計算延遲所需的工時窗。
3. **門檻數值**：I1–I4 各 severity 映射門檻、tripwire 硬上限的實際值。
4. **議題抽取模型**：採用哪個 LLM、跑批頻率與觸發條件。
5. **PDPA 合規**：蒐集處理目的、告知同意機制。
6. **CRM 串接可能性**：是否能取得下游客訴/流失資料作 weak label。
